Early Beta Warning

This firmware controls reloading equipment. Verify every charge with a calibrated scale, supervise all motor activity, and keep a known-good UF2 available for rollback.

Pico 2 W Only

This ML beta does not support Pico W / RP2040. The AI, REST, OTA, web portal, scale-driver, and runtime-learning feature set exceeds the RP2040 256 KB RAM budget; RP2350 provides 512 KB.

Public Beta Release

This release is public so builders can inspect the source, download the UF2, and test carefully. Treat it as early beta firmware until more powder validation is complete.

What changed

Different From Original OpenTrickler Firmware

This branch keeps the normal OpenTrickler charge flow, display workflow, scale support, and motor control foundation, then adds a profile model that can be characterized, calibrated, saved, and guarded by recent production evidence without drifting the characterized machine parameters.

Profile AI model

Each profile can store learned flow, tail, confidence, machine timing, and recovery behavior.

Flow characterization

Measures coarse, trim, fine, and micro-recovery delivery to build a model for the selected setup.

Machine calibration

Captures scale response, settle behavior, open-loop flow, and handoff margins after a model exists.

Runtime learning

Normal throws maintain a rolling phase-specific safety window without rewriting characterization.

Automatic safety governor

Recent phase results may widen guards immediately; only stable, accurate history may tighten them.

Measured recovery

Beta 11 calculates bounded approach and micro doses from observed effective recovery flow.

REST and OTA

Status, tuning, runtime history, charge state, system info, and OTA staging are available over Wi-Fi.

Full tuning reference

Characterization, calibration, saved-model fields, runtime observations, and REST parameters are documented.

Compatibility

Hardware And Scale Support

The codebase builds only for Raspberry Pi Pico 2 W / RP2350 and targets OpenTrickler controller hardware with stepper-driven coarse/fine tricklers, optional servo gate, NeoPixel status, and the mini 12864 display. Pico W / RP2040 is intentionally disabled because the firmware no longer fits within its 256 KB RAM budget.

This firmware pairs especially well with the ultra-low-flow small fine tube from the original Eamars OpenTrickler hardware repository: Ultra Low Flow Fine Trickler Tube - Scaled to 99.7 Percent.stl. Its slower, smoother feed gives the ML controller far more precision during final approach, recovery, and micro_heal. Other fine tube designs may work, but re-characterize and rerun machine calibration after changing tube geometry.

  • 2.4 GHz Wi-Fi only, with AP mode for first-time setup.
  • Supported scale drivers include AND FX-i, Steinberg SBS, G&G/JJB, US Solid JFDBS, JM Science, Creedmoor, Radwag PS R2, Sartorius, Generic, and Ohaus PR.
  • Scale serial formats include 4800, 9600, and 19200 baud with 8N1 or 7N1 framing.
  • Use a separate profile when powder, tube setup, scale, gate, target range, or mechanical behavior changes.

Get running

Install And First Boot

Use USB flashing for first install, recovery, and rollback. Use OTA only after the device is already running firmware that supports OTA staging.

3

Copy UF2

Copy app.uf2 to the RPI-RP2 drive. The board disconnects and reboots automatically.

4

Verify version

Open the web portal or REST system information and confirm the expected release string.

Network setup

Connect To Wi-Fi

On first boot, the controller starts its own access point. Configure a 2.4 GHz network from a laptop or phone, then use the assigned LAN address for the web portal, REST API, and OTA.

AP mode Look for an OpenTrickler SSID with a device-specific suffix. Default password: opentrickler.
Setup portal After joining the AP, browse to http://192.168.4.1 and enter 2.4 GHz Wi-Fi credentials.
Station mode After reboot, the display or portal shows the LAN address. The onboard LED is solid when connected.

Profiles first

Create The Right Profile

Profiles are the boundary around learned behavior. Keep separate profiles for different powders, tube geometries, scale setups, gate behavior, and target ranges whenever practical.

  1. Open the web portal or local controller UI.
  2. Go to Settings and choose a profile slot.
  3. Name the profile for the powder/setup, such as Varget 36.5 or H4350 short tube.
  4. Copy or enter trusted baseline motor settings.
  5. Save the profile before characterization.
  6. Confirm the selected profile before every production run.

Use a new profile when

Powder, tube, scale, target range, gate, or mechanical behavior changes enough to affect flow.

Re-characterize when

The saved model no longer matches real throws, or after meaningful hardware or scale changes.

AI tuning workflow

Run Characterization

Characterization measures how this machine, this powder, this tube setup, and this scale behave. It builds a working model and waits for you to save it to the selected profile.

1. Prepare

Fill the hopper, confirm the tube and gate setup, connect the scale, warm up the scale, and verify zero.

2. Select profile

Open Settings, AI Tuning, select the correct profile, and enter the normal target charge weight.

3. Start

Start characterization and follow the normal remove/return pan prompts without changing the setup.

4. Sample

The firmware samples coarse, trim, fine, and recovery behavior while tracking delivered weight and tail.

5. Review

Wait for the tuner state to reach ready_to_save; cancel and rerun if the scale or powder flow was unstable.

6. Save

Apply/save the model to the selected profile. The model persists in flash and becomes available for production.

Good characterization inputs

Stable scale, repeatable pan handling, clean tube flow, correct target weight, and the same setup you plan to use for production.

Discard and rerun if

The scale drifts, powder bridges, the wrong profile is selected, the pan workflow is interrupted, or the model obviously learned from bad samples.

Stage Motor Default plan What the firmware records
characterizing_coarseCoarse only12 samples, 120 gn budget, 8.0 gn nominal sample targetSpeed, motor-on time, delivered weight, tail after stop, flow, best bulk speed, and trim-speed candidate.
characterizing_fineFine only12 samples, 36 gn budget, 1.75 gn nominal sample targetFine flow, fine tail, fast-finish behavior, first four recovery/micro samples, and fine-window recommendation.

The first four fine samples are low-speed recovery samples near 0.08, 0.12, 0.20, and 0.35 rps. Full parameter details are in the AI tuning reference.

Machine timing

Run Machine Calibration

Calibration should be run after a characterization model has been saved. It measures the machine behavior that affects handoff timing and final recovery.

1

Save model first

Machine calibration requires an existing saved characterization model for the selected profile.

2

Keep setup fixed

Use the same powder, tube, scale, pan, and target range as characterization.

3

Start calibration

Start from the AI Tuning panel and let the controller complete the planned timing samples.

4

Save and verify

Save the completed calibration, then run supervised production charges before relying on it.

What it measures

Scale sample period, response latency, settle time, coarse tail, fine tail, open-loop flow, and bulk handoff margin.

When to rerun

After changing scale, tube, powder behavior, gate geometry, major motor settings, or when handoff and recovery feel wrong.

Stage Motor Default plan What changes in the saved model
calibrating_coarseCoarse only8 samples using production and trim speeds, with 250-850 ms pulsesScale response, coarse settle, coarse tail average/high value, coarse uncertainty, bulk flow, trim flow, and handoff margin.
calibrating_fineFine only8 samples using fine and recovery speeds, with 500-7000 ms pulsesFine response, fine settle, fine tail average/high value, fine uncertainty, fine flow, micro-flow, and post-finish watch time.

Calibration requires a saved characterization model and at least three valid coarse plus three valid fine samples. See the AI tuning reference for all saved fields and REST parameters.

Production evidence

Automatic Runtime Learning

Runtime learning happens during normal throws. Beta 11 evaluates complete coarse settling, fast-finish tail, and recovery outcomes separately, then changes temporary guards without rewriting the characterized speeds, flows, or machine-calibration fields.

  • Unstable or overthrow-prone history can only make a phase safer.
  • Stable, accurate history is required before a phase can tighten.
  • Near-target recovery waits for passive powder arrival before running the motor.
  • Recovery doses use measured effective flow and escalate after verified no progress.
  • Re-characterize if the setup no longer behaves like the saved model.
Recent evidence Automatic response
Stable and accuratePercentile guards may tighten within fixed safety bounds.
Fast-finish overthrowsUse the complete-tail P90 floor and block tightening.
Recovery overthrowsWait for tail drain and calculate bounded doses from measured recovery flow.
Late coarse landingKeep powered coarse top-up disabled and observe the complete passive tail.

Daily operation

Throwing Charges

  1. Select the correct profile.
  2. Enter the target charge weight in grains.
  3. Place the pan on the scale and wait for zero.
  4. Let the controller charge, settle, recover if needed, and request cup removal.
  5. Remove the pan only when prompted.
  6. Verify the charge independently before use.
  7. Return the pan for the next cycle.

Developer setup

Build From Source

Use the Raspberry Pi Pico SDK toolchain, CMake, Ninja, Git, Python 3, and repository submodules. Pico 2 W / RP2350 is the only supported beta target. A Pico W / RP2040 configure attempt fails immediately because the current firmware exceeds the RP2040 RAM budget.

Clone and submodules

git clone <repository-url-from-the-github-code-button>
cd Opentrickler_ML
git submodule update --init --recursive

Configure Pico 2 W release

.\configure_env.ps1
cmake -B build-pico2w-release -G Ninja -DCMAKE_BUILD_TYPE=Release -DPICO_BOARD=pico2_w

Build firmware

cmake --build build-pico2w-release --config Release

Build outputs are ignored by git. Use app.uf2 for USB flashing and app.bin for OTA upload.

Release status

2026.07.12-beta.11

Beta 11 responds to the first beta10 powder session by removing powered coarse top-up, separating runtime guards from characterization, and replacing repeated recovery stalls with measured doses.

Changes

  • Removed powered coarse top-up from production throws.
  • Added a complete motor-off coarse_settle observation before fine control.
  • Stopped production throws from rewriting characterization and calibration fields.
  • Added measured recovery-flow P25 and median statistics.
  • Replaced repeated micro-heal stalls with three-zone bounded recovery doses.

Expected behavior

  • Characterized coarse and fine speeds remain unchanged.
  • Residual bulk powder drains before the controller decides how much fine work remains.
  • Runtime percentiles change temporary guards, not the saved physical model.
  • No-progress recovery pulses escalate within strict pulse and time limits.

Test status

  • Pico 2 W release build passes locally.
  • The 29-throw beta10 RL17 session identifies coarse top-up as the dominant fast-overthrow source.
  • The latest runtime window provides 16 usable recovery-flow measurements.
  • Static, replay, collector, and OTA checks precede supervised powder verification.
