Complete beta reference

Characterization and calibration are the core difference in this firmware.

The ML layer does not try to make every OpenTrickler use one fixed set of motor numbers. It learns a saved model per profile, then uses runtime observations to keep that model aligned with the real machine.

User workflow

Recommended Order

Use this order for each powder, tube, scale, pan, and target-range combination. A profile is the boundary around the learned model, so changing the physical setup should usually mean a new profile or a rerun.

1. Create profile

Name the profile for the powder/setup and enter trusted baseline motor settings.

2. Characterize

Run controlled coarse and fine samples so the firmware can learn flow, tail, and recovery behavior.

3. Save model

Apply the working model only after the tuner reaches ready_to_save and the samples were clean.

4. Calibrate machine

Measure scale lag, settle timing, open-loop flow, and safe handoff margins using the saved model.

5. Save calibration

Apply again so the profile stores both powder characterization and machine timing data.

6. Verify throws

Run supervised production charges and verify every charge while runtime evidence establishes its safety window.

First learning pass

How Characterization Works

Characterization starts a normal charge-mode workflow from the web portal, but the AI planner supplies each motor pulse. The firmware measures the scale before the pulse, immediately after the motor stops, and after the powder/scale response settles. Every sample becomes a flow sample with speed, motor-on time, delivered weight, tail, and flow.

Stage Motor mode Default limit Speed selection Stop condition
characterizing_coarse coarse 12 samples or 120 gn total budget Sweeps between profile coarse min/max speed, capped by motor and hardware safety limits. Stops when coarse_sample_count is reached or coarse_budget_gn is used.
characterizing_fine fine 12 samples or 36 gn total budget Sweeps between profile fine min/max speed. First four samples are low-speed recovery probes. Stops when fine_sample_count is reached or fine_budget_gn is used.

Coarse output

Coarse flow slope/intercept, average/guarded coarse tail, best bulk speed, best bulk flow, trim speed, trim flow, and trim tail.

Fine output

Fine flow slope/intercept, fine tail, best fine speed, best fine flow, recovery speed, recovery flow, recovery tail, fine window, and fine tube profile.

The first four fine recovery samples use approximately 0.08, 0.12, 0.20, and 0.35 rps with long motor-on windows of about 12000, 10000, 8000, and 6500 ms. This is what teaches the firmware how the fine tube behaves near target during fine_recover and micro_heal.

Second learning pass

How Machine Calibration Works

Machine calibration requires an existing saved characterization model. It copies that saved model into a working session, clears the machine-calibration portion, then runs timing samples at the speeds the saved model expects to use in production.

Stage Motor mode Default plan What is measured
calibrating_coarse coarse 8 samples. Pulses are 250, 450, 650, and 850 ms at production coarse speed, then repeated at trim speed. Scale period, first response, settle time, coarse tail average/high value, uncertainty, bulk flow, trim flow, and recommended bulk handoff margin.
calibrating_fine fine 8 samples. Pulses are 500, 800, 1100, and 1400 ms at fine speed, then 2500, 3500, 5000, and 7000 ms at recovery speed. Fine response, fine settle, fine tail average/high value, fine uncertainty, fine flow, micro flow, recommended trim stop, and post-finish watch time.

Minimum valid data

Calibration must capture at least 3 valid coarse samples and 3 valid fine samples. It can stop after that if the stage budget guard has been reached.

Why it matters

Characterization learns powder flow. Calibration learns the actual machine delay and tail risk, which changes safe bulk handoff and final recovery behavior.
